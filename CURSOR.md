# AI Layer — оркестратор

## Для чего этот файл
Это **единая точка входа** и главная карта для агента и разработчика. Перед **любой** задачей (даже мелкой) прочитай его (или нужный раздел). Он заменяет длинные повторяющиеся инструкции в чате, экономит токены и делает поведение AI предсказуемым и проектно-специфичным.

## Как начинать любую задачу в Cursor (обязательный порядок)

1. **Прочитай этот файл**
2. **Изучи Живую Документацию** проекта (в этом шаблоне — README.md + все `.md` в корне и `ai-layer-builder/`. В реальных проектах — `@PROJECT_KNOWLEDGE.md`). Цель: понять текущее состояние архитектуры, правил, известных проблем.
3. **Обязательно составь/обнови План** (создай или отредактируй `@PLAN.md` или файл в `.cursor/plans/`). **Никогда не начинай писать код без плана** — это дороже переделок.
4. **Подключай контекст точечно**: `@путь/к/файлу` или `@папка/`. Никогда не тащи весь репозиторий. Используй `.cursorignore` для тяжёлых/секретных путей.
5. **Для типовых задач** — используй таблицу маршрутизации ниже или готовые slash-команды из `.cursor/commands/`.
6. **Цикл принятия решения агентом** (выполняй всегда):
   - Классифицируй задачу по таблице маршрутизации.
   - Прочитай живую документацию и релевантные правила.
   - Составь короткий План (в PLAN.md) — на быстрой модели.
   - Открой только нужные `@` файлы + правила.
   - При необходимости вызови skill **slash-командой** из реестра (см. `.cursor/rules/project-skills-registry.mdc`).
   - Выполни работу по шагам.
   - **Структурно обнови Живую Документацию** (добавь/измени разделы: Архитектура, Правила валидации, Известные проблемы, Текущий стек — **не** append-only лог "я добавил файл X, починил Y").
   - Проверь критерии "Готово" из команды/плана.
   - Заверши.

**Ключевые практики экономии и качества**:
- Перед любым тяжёлым Agent-прогоном — **Plan Mode** (Shift+Tab).
- Для больших задач — новый чат + ссылка на `@CURSOR.md` (длинная история = дорого).
- Повторяющиеся сценарии выноси в commands/skills, не расписывай заново.
- preCompact / compaction: сжимай вывод, не раздувай дифы.

## Политика токенов и экономии контекста

Подробно: `@.cursor/rules/token-economy.mdc`

Кратко:
- По умолчанию **Auto / быстрая модель** (типовая работа, локальные правки, поиск).
- Сильная модель — только после короткого плана на быстрой + явного переключения (для неоднозначной архитектуры, критичной безопасности, глубокой отладки).
- **Главный расход** — не длина промпта, а контекст (файлы + история чата + tool calls). Береги окно!
- Практики: точечный `@`, `.cursorignore`, новый чат на новую задачу, короткие итерации, preCompact при заполнении контекста, вынос повторяющихся сценариев в skills/commands.
- Перед тяжёлым Agent — **Plan Mode** (дешевле переделок).
- Мелкие правки — Ask/Inline/Cmd+K вместо полного Agent.

## Живая Документация и Планирование (самое важное правило)

- **Живая Документация (State/Memory)**: В реальных проектах — `PROJECT_KNOWLEDGE.md` (или папка `docs/`). В этом шаблоне — сами файлы + README.
  - **Обязательно** обновлять **структурно** после каждой задачи: добавлять/менять разделы "Архитектура", "Бизнес-правила", "Текущий стек", "Известные проблемы", "Точки входа".
  - **Запрещено** вести append-only лог ("я добавил файл X", "починил баг Y", "обновил дату"). Документация должна показывать **"как код работает сейчас"**, чтобы новый чат мог мгновенно войти в курс без перечитывания всей истории.
- **Планирование (Plan-Driven Development)**: Любая разработка (фича, багфикс, рефакторинг, даже мелкий) **обязана начинаться с плана** в `PLAN.md` (или `.cursor/plans/`). План обновляется по мере выполнения. Это фундаментально дешевле и качественнее, чем "просто кодить".

## Обновление AI-слоя

Обновляй при:
- Смене стека / крупном рефакторинге / "разъезде" конфигурации.
- Добавлении новых команд, skills, правил.
- Накоплении хаоса в `.cursor/` или росте контекста.
- После каждого bootstrap нового проекта.

**Как обновить**: Открой `@ai-layer-builder/ai-builder.md`, пройди шаги 0-8 строго по порядку, обнови дату в шапке этого файла и при необходимости другие файлы.

**Антипаттерны (что не делать)**:
- Вставлять полные тексты правил из `.mdc` в этот файл.
- Описывать весь стек на 3 экрана — только границы + ссылки.
- Начинать код без плана.
- Обновлять документацию логами вместо структуры.
- Тянуть тяжёлые файлы без `.cursorignore`.
- Выдумывать несуществующие скилы/команды.
