# /feature — Новая фича от ТЗ до PR

Файл в `.cursor/commands/feature.md` — текст, который подставляется в чат при выборе `/feature`.

## Цель
Реализовать новую фичу от понимания ТЗ до рабочего кода + тестов + обновлённой Живой Документации + готовности к PR. Без хаоса и переделок.

## Входы (что пользователь должен указать)
- Краткое описание фичи или ссылка на issue/TЗ.
- (Опционально) Скриншоты, ветка, существующий код для примера.
- Приоритет и acceptance criteria (если есть).

## Шаги для агента (обязательный порядок)

1. **Составить План** (создать/обновить `@PLAN.md` или `.cursor/plans/feature-xxx.md`).
   - Разбить фичу на подзадачи.
   - Определить затронутые файлы/модули.
   - Указать риски и как их минимизировать.
   - Описать, как будет выглядеть обновлённая Живая Документация.

2. **Изучить контекст**:
   - Прочитать релевантные части Живой Документации (`@PROJECT_KNOWLEDGE.md` или эквивалент).
   - Открыть существующий код в затронутых модулях (точечный `@`).
   - При необходимости — `/review` существующий код.

3. **Реализовать** по плану:
   - Писать код + тесты (unit + integration, если применимо).
   - Следовать правилам из `.cursor/rules/stack-specific.mdc` и `architecture.mdc`.
   - Делать маленькие коммиты/дифы (не огромный PR за раз).

4. **Протестировать**:
   - Запустить тесты.
   - Проверить edge cases.
   - Если есть UI — проверить вручную или через e2e.

5. **Обновить Живую Документацию** (структурно!):
   - Добавить/изменить разделы: Архитектура (новые компоненты/потоки), Правила валидации, Известные проблемы, Точки входа.
   - Не писать "я добавил фичу X" — описать, как теперь работает система.

6. **Подготовить к PR**:
   - Проверить линтеры/форматтеры.
   - Обновить README/CHANGELOG при необходимости.
   - Создать описание PR с ссылкой на план и изменения в документации.

## Ограничения
- Не трогать модули, не указанные в плане (без явного разрешения).
- Лимит дифа: один PR — не больше 500-800 строк изменений (разбивай на несколько).
- Тесты обязательны для новой бизнес-логики и публичных API.
- Не использовать сильную модель без плана на быстрой.

## Готово (критерий завершения)
- Код работает, тесты зелёные.
- Живая Документация обновлена структурно (отражает текущее состояние).
- План в PLAN.md закрыт/отмечен выполненным.
- Готово к review / PR.
- **Обязательно**: обновлён `@PROJECT_KNOWLEDGE.md` (или эквивалент) в соответствии с изменениями (без добавления логов).
