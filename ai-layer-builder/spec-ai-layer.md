# Спецификация корневого `CURSOR.md`

Файл в **корне репозитория**: `CURSOR.md`. Это **оркестратор**: куда смотреть агенту и человеку перед задачей. Не дублировать длинные правила — только ссылки и таблицы.

## Обязательные секции

### 1. Метаданные

- Одна строка: название/назначение проекта.
- `Последнее обновление AI-слоя: YYYY-MM-DD` (проставлять при каждом обновлении конфигурации).

### 2. Как начинать любую задачу в Cursor

- Сначала прочитать этот файл (или релевантный раздел).
- Изучить **Живую Документацию** проекта (например, `@PROJECT_KNOWLEDGE.md`), чтобы понять текущее состояние архитектуры и логики.
- Обязательно составить/обновить **План** (например, `@PLAN.md`) перед написанием кода.
- Подключать контекст точечно: `@путь/к/файлу`, не весь репозиторий без нужды.
- Для типовых действий: таблица «задача → команда или skill».
- Кратко **цикл принятия решения агентом**: классификация задачи по таблице → прочитать живую документацию → составить план → открыть нущие `@` → написать код → **обновить живую документацию** (не логом, а структурно) → завершить.

### 3. Карта репозитория (кратко)

- 5–15 строк: корневые пакеты, где API/UI/домен, где тесты, где конфиги.
- Если monorepo — подпакеты одной строкой каждый.

### 4. Маршрутизация

Таблица в markdown, например:

| Задача | Действие |
|--------|----------|
| Новая фича или баг | Создать План, обновить код, обновить Живую Документацию |
| Вызов команды | Вызвать соответствующий сценарий из `.cursor/commands/` |
| Использование навыка | Использовать соответствующий навык из `.cursor/skills/` |
| … | … |

### 5. Ссылки на AI-слой

- `@.cursor/rules/` — перечисли **имена** ключевых `.mdc` и одну фразу зачем каждый.
- Также здесь может указываться правило **реестра глобальных скилов** (напр. `global-skills-routing.mdc` / `project-skills-registry.mdc`) — **обязательно** одна строка: это **источник истины**, какие глобальные `SKILL.md` читать в каких сценариях; не дублируй полный список скилов в `CURSOR.md`, только ссылка `@…`.
- `@.cursor/commands/` — только **проектные** slash-команды (`*.md`); глобальные `/create-rule` и т.п. в репозиторий не кладут.
- `@.cursor/skills/` — перечень папок skills и когда вызывать.

### 6. Политика токенов (один абзац)

- Ссылка: `@.cursor/rules/token-economy.mdc`.
- Одно предложение: быстрая модель по умолчанию, сильная — по эскалации.

### 6.5. Живая Документация и Планирование

- Ссылка: Указать путь на файл(ы) структурированной документации (напр., `@PROJECT_KNOWLEDGE.md`).
- Правило ведения: Документация не является append-only логом событий (запрещено логировать "я поправил баг"), она должна описывать текущую структуру проекта и обновляться агентом в нужных разделах после каждой задачи.
- Ссылка: Указать путь на план (напр., `@PLAN.md`) и правило планирования каждой фичи перед началом кодинга.

### 7. Обновление слоя

- Когда обновлять слой: смена стека, крупный рефакторинг, новая команда, накопленный хаос в `.cursor/`.
- Указание: регулярно поддерживать этот файл и папки `.cursor/` в актуальном состоянии.

### 8. Опционально

- `AGENTS.md`: если используется в команде — одна строка «не дублировать длинные списки из ai-layer».

## Антипаттерны

- Вставлять полные тексты правил из `.mdc` в `CURSOR.md`.
- Описывать весь стек на несколько экранов — только границы и ссылки.
