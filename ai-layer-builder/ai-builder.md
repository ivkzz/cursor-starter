# AI Layer Bootstrap — оркестратор `.cursor/` и корневого `ai-layer.md`

Ты не «натягиваешь шаблон», а **проектируешь управляемую автономию**: пользователь задаёт **намерение** (slash-команды, `@ai-layer.md`, короткий запрос), а **AI-слой** (корневой оркестратор + правила + команды + скилы) **классифицирует задачу, выбирает маршрут, принимает решения в рамках политик** и только при необходимости уточняет у человека.

## Роль AI-слоя (что ты должен явно спроектировать)

1. **`ai-layer.md` в корне** — единственная **карта маршрутизации**: куда смотреть, какую команду/skill выбрать, когда эскалировать модель или план.
2. **`.cursor/rules/*.mdc`** — **жёсткие политики для ИИ** (токены, язык, архитектурные границы, стек, **какие скилы обязаны читаться в каких сценариях**). Правила — не декорация, а **оркестратор поведения агента**.
3. **`.cursor/commands/*.md`** — **контракты намерения**: пользователь выбирает `/feature`, `/fix-bug` и т.д.; внутри — чеклист, входы, критерий «готово». Агент **сам** раскладывает работу по шагам, не ожидая микроменеджмента.
4. **`.cursor/skills/`** — повторяемые **длинные** сценарии команды/продукта; глобальные скилы на машине разработчика (см. справку ниже) — в реестре шага 2.5; проектные — рядом в репо.
5. **Hooks** — только если дают измеримый выигрыш (события, автопроверки), без шума.

Итог: **пользователь указывает что сделать → слой решает как** (в пределах правил и таблиц маршрутизации).

## Инварианты

1. **Оркестратор-индекс** — только `ai-layer.md` в корне. Кратко: таблицы, `@`-ссылки; **не** копируй длинные тексты правил внутрь него.
2. **Rules / Hooks / Skills** — форматы и создание через встроенные **`/create-rule`**, **`/create-hook`**, **`/create-skill`**. Проектные шаблоны в `.cursor/commands/` — только **свои** сценарии (`spec-commands.md`), не дубли `create-*`.
3. **Merge, не выжигай**: существующий `.cursor/`, `hooks.json`, `ai-layer.md` — проанализируй, дополни, устаревшее поправь; полная перезапись только по явной просьбе пользователя.
4. **Токены**: после шага 0 не повторяй длинные политики — `@.cursor/rules/token-economy.mdc`.
5. **Креативность с дисциплиной**: предлагай **минимальный, но достаточный** набор правил/команд/skill-роутинга под реальный репозиторий; избегай «энциклопедий» и дублирования `ai-layer` ↔ rules.

Структуру `ai-layer.md` возьми из `@ai-layer-builder/spec-ai-layer.md`. Команды (не create-*) — `@ai-layer-builder/spec-commands.md`.

После каждого крупного шага дай краткий отчёт.

### Справка: глобальные скилы (только для человека; агент **не** устанавливает)

На машине разработчика **уже предполагается** установленный набор скилов для Cursor (web / full-stack / testing), чтобы шаг 2.5 строил реестр по реальным путям. Установку выполняет **только человек** вне этого сценария; **не запускай** `npx`, установщики и сетевые команды ради скилов.

Для справки (копирует пользователь сам при настройке окружения):

```bash
npx antigravity-awesome-skills --cursor --bundle web-wizard,full-stack-essentials,testing-pro
```

Если на диске скилов нет или структура другая — реестр в шаге 2.5 строй **только** из того, что реально найдено; bootstrap не блокируй.

## Шаг 0. Token economy (обязательно первым)

Проверь наличие **`.cursorignore`** в корне (или создай): исключай тяжёлые артефакты, локи, секреты — иначе индексация съедает контекст и кредиты.

Создай или обнови `.cursor/rules/token-economy.mdc` (ориентир — уже лежащий в репозитории; расширяй под проект):

- Режим **Auto** по умолчанию.
- **Модели**: не перечисляй устаревающий список имён — раздели на **быструю** и **сильную**; для сильной — сначала план на быстрой, затем явное переключение; актуальные имена — из UI Cursor.
- Экономия: точечный `@`; короткие ответы; не копируй текст правил в чат; короткие сессии; при росте контекста — резюме и `@ai-layer.md`.
- **preCompact** / compaction: при заполнении контекста сжимать вывод, не раздувать дифы без нужды.

## Шаг 1. Режим проекта

**A. Greenfield (почти пусто)**  
Нет осмысленного кода или манифестов: зафиксируй у пользователя цели, стек, язык ответов, тесты/CI, стиль коммитов. Код приложения генерируй **только если пользователь явно просит** — иначе ограничься AI-слоем и **маршрутизацией автономии** (команды + правила).

**B. Brownfield (уже есть проект)**  
Инвентаризация: дерево каталогов, README, package/lock, конфиги, CI, тесты, точки входа. Прочитай существующий `.cursor/`, `ai-layer.md`, **если есть** `.cursor/rules/*skills*` — не дублируй, обнови дату/состав при необходимости.

## Шаг 2. Критический разбор и дизайн оркестрации (коротко, письменно)

Ответь компактно:

- Что мешает AI работать дёшево и качественно (размер файлов, тесты, дубли, границы модулей)?
- **Цикл автономии**: какие типы задач агент распознаёт сам; что всегда требует уточнения у человека; где таблица в `ai-layer.md` vs отдельное правило.
- Какие **rules** (имена `.mdc`) реально нужны под этот стек?
- Какие **hooks** (события + зачем) — только осмысленные.
- Какие **skills** (глобальные по реестру + проектные в `.cursor/skills/`) и **команды** в `.cursor/commands/` (`spec-commands.md`).

## Шаг 2.5. Реестр глобальных скилов → правило (один раз на проект; при повторном bootstrap — обновление)

Цель: чтобы агент **не гадал**, какие установленные **глобальные** скилы существуют, а **следовал явному реестру в правиле**.

1. **Обнаружение** (выполни по возможности через терминал и/или поиск по файловой системе пользователя):
   - Типичные корни Cursor на Windows: `%USERPROFILE%\.cursor\skills-cursor\`, `%USERPROFILE%\.cursor\plugins\` (вложенные `skills\**\SKILL.md`).
   - На macOS/Linux: `~/.cursor/skills-cursor/`, `~/.cursor/plugins/`.
   - Собери список каталогов/имён, у которых есть `SKILL.md` (достаточно **относительного пути от корня `.cursor`** или полного пути — выбери один стиль и держи его в правиле).

2. **Отбор под проект**: из найденного отметь скилы, релевантные **этому** репозиторию (стек, домен, CI). Остальные можно перечислить кратко как «есть, не приоритет».

3. **Создание правила** через **`/create-rule`** (предпочтительно) или вручную файл `.mdc` в `.cursor/rules/`:
   - Имя напр. **`global-skills-routing.mdc`** или **`project-skills-registry.mdc`**.
   - **`alwaysApply: true`** (или узкий `globs`, если реестр строго привязан к типам файлов — по умолчанию лучше always для маршрутизации скилов).
   - Содержимое правила минимум:
     - Таблица: **сценарий / задача** → **какой skill читать первым** (полный путь к `SKILL.md` или однозначная инструкция «найди в `…/skills-cursor/…`»).
     - Строка: **«реестр обновляется при повторном `@ai-layer-builder/ai-builder.md` или при смене бандла скилов»**.
     - Запрет «выдумывать несуществующие скилы»: только из отсканированного списка + проектные `.cursor/skills/`.

4. Если правило с реестром **уже есть** — смержи изменения (новые пути, удалённые скилы), не плоди второй файл с той же ролью.

5. Добавь в **`ai-layer.md`** (шаг 7) одну строку в секции ссылок: указание на это правило как на **источник истины по глобальным скилам**.

## Шаг 3. Rules

Запусти **`/create-rule`** (или вручную `.mdc`, если команда недоступна) и дополни `.cursor/rules/`: минимум `token-economy.mdc`, `core.mdc`; при необходимости `architecture.mdc`, `stack-specific.mdc` и узкие файлы. Один фокус на файл, короткий текст.

Убедись, что **`core.mdc`** (или отдельное правило) явно говорит: **перед нетривиальной задачей — `@ai-layer.md`**, затем при необходимости **skill из реестра шага 2.5**.

---

## Шаг 4. Hooks

Запусти **`/create-hook`** и добавь только полезные записи в `hooks.json` и скрипты в `.cursor/hooks/`.

---

## Шаг 5. Skills

Запусти **`/create-skill`**. Проектные скилы — `.cursor/skills/<name>/SKILL.md`.  
Не дублируй глобальные сценарии в проекте, если достаточно **реестра из шага 2.5** + тонкого проектного skill-обёртки.

---

## Шаг 6. Прочие команды

По `@ai-layer-builder/spec-commands.md` добавь или обнови `.md` в `.cursor/commands/`. Каждая команда должна давать агенту **достаточно структуры для самостоятельного плана** (входы → шаги → критерий готово → ограничения).

---

## Шаг 7. Оркестратор `ai-layer.md`

Создай или обнови **`ai-layer.md` в корне** по `@ai-layer-builder/spec-ai-layer.md`:

- Карта репо, маршрутизация, ссылки на rules/commands/skills.
- Явный блок **«Как агент принимает решение»**: прочитал оркестратор → выбрал строку таблицы → открыл нужные `@` → применил skill из реестра → зафиксировал допущения.
- Дата последнего bootstrap, как обновлять слой.

---

## Шаг 8. Финализация

1. Дерево `.cursor/` (кратко).
2. Список изменений и зачем они этому проекту.
3. Подтверди: создан/обновлён ли **реестр скилов (2.5)** (и при отсутствии глобальных скилов на диске — что зафиксировано в реестре/отчёте).
4. Предложи проверить 1–2 hook/skill вручную.
5. Подтверди применение token-economy.

Начни строго по порядку: **0 → 1 → 2 → 2.5 → 3 → 4 → 5 → 6 → 7 → 8**. Установку глобальных скилов не предлагай и не выполняй — см. справку выше.
